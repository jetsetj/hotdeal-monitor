# 에이전트 팀 운영 가이드

## 팀 구성

| 에이전트 | 역할 | 입력 | 출력 |
|----------|------|------|------|
| **Planner** | 기획 | 사용자 요구사항 | SPEC.md |
| **Developer** | 개발 | SPEC.md | 완전한 코드 |
| **Reviewer** | 검수 | 코드 + SPEC.md | REVIEW.md |

## 작업 흐름

```
┌─────────────┐
│  사용자 요청  │
└──────┬──────┘
       ▼
┌─────────────┐     ┌─────────────┐
│  Planner     │────▶│  SPEC.md    │
│  (기획)      │     │  작성       │
└──────┬──────┘     └──────┬──────┘
       │                  │
       ▼                  ▼
┌─────────────┐     ┌─────────────┐
│  Developer   │◀────│  사양 전달   │
│  (개발)      │     │            │
└──────┬──────┘     └─────────────┘
       │
       ▼
┌─────────────┐
│  코드 완성   │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────┐
│  Reviewer    │────▶│  REVIEW.md  │
│  (검수)      │     │  작성       │
└──────┬──────┘     └─────────────┘
       │
       ▼
┌─────────────┐
│  수정 요청   │ (문제 발견 시 Developer에게)
│  또는 승인   │
└─────────────┘
```

## 규칙

### 1. Planner 규칙
- 사용자의 모호한 요구를 구체화할 것
- 기술 선택의 이유를 설명할 것
- 예시 RSS 사이트 목록 제공

### 2. Developer 규칙
- SPEC.md의 모든 항목 구현 필수
- 코딩 컨벤션 준수 (ESLint, Prettier)
- 예상 질문 사항을 미리 정리
- 환경변수 설정 가이드 포함

### 3. Reviewer 규칙
- 치명적 버그는 즉시 수정 요청
- 승인/거부 이유를 명확히 기록
- 통과 기준을 투명하게 공개

## 사용 방법

### 에이전트 실행
```bash
# Planner 에이전트 시작
/task "planner.md의 지침에 따라 새로운 웹 앱 기획"

# Developer 에이전트 시작  
/task "developer.md의 지침에 따라 SPEC.md 기반으로 개발"

# Reviewer 에이전트 시작
/task "reviewer.md의 지침에 따라 코드 검수"
```

### 검수 통과 기준
모든 Major + Critical 항목 통과 시 승인

### 반복 주기
검수发现问题 → Developer 수정 → Reviewer 재검토 → 통과
